Documentation is critical but time-consuming. AI can transform code, APIs, processes, and systems into clear, comprehensive documentation. This lesson teaches you how to generate user manuals, API docs, code comments, and technical guides that actually help people.

Why AI for Documentation?

Documentation Challenges AI Solves:

  • Time-sink: Docs are always last priority—AI makes it fast
  • Out-of-date: Generate fresh docs from current code/systems
  • Inconsistency: Maintain uniform structure and style
  • Audience mismatch: Generate versions for different skill levels
  • Language barriers: Translate technical concepts clearly

Application 1: Code Documentation

Inline Code Comments

Prompt Template:

"Add comprehensive inline comments to this code. For each function:
- Purpose and what it returns
- Parameters explained
- Any non-obvious logic
- Edge cases handled

Follow [language] documentation standards (e.g., JSDoc for JavaScript, docstrings for Python).

[Paste code here]"

Function/Class Documentation

Example: Python Function

[paste function code]

Include:

Application 2: API Documentation

Endpoint Documentation

Comprehensive API Doc Prompt:

"Generate API documentation for this endpoint. Format as markdown:

Endpoint: [e.g., POST /api/users]
Purpose: [brief description]

Include:
1. Overview: What this endpoint does and when to use it
2. Authentication: Required auth method
3. Request:
- Headers (table format)
- Body parameters (table: name, type, required?, description)
- Example request (JSON)
4. Response:
- Success response (200) with example JSON
- Error responses (400, 401, 404, 500) with examples
5. Code Examples: cURL, Python requests, JavaScript fetch
6. Rate Limits: Any throttling
7. Notes: Edge cases, best practices"

Full API Reference Generation

"I have a REST API with 15 endpoints. Generate a complete API reference guide. For each endpoint, provide: HTTP method, path, description, auth requirements, request/response schemas, example calls. Structure: Introduction → Authentication section → Endpoints (grouped by resource: Users, Products, Orders) → Error codes reference → Rate limiting section. Format as markdown suitable for static site generator."

Application 3: User Manuals & Guides

Software User Manual

User Manual Prompt Structure:

"Create a user manual for [software/app]. Target audience: [non-technical users / admins / developers].

Structure:

  1. Introduction: What is it? Who is it for? Key benefits
  2. Getting Started: Installation, account setup, first steps
  3. Core Features: Step-by-step guides for main tasks (include screenshots placeholders)
  4. Advanced Features: Optional but powerful features
  5. Troubleshooting: Common problems and solutions
  6. FAQ: Top 10 questions
  7. Glossary: Technical terms explained

Tone: Clear, patient, encouraging. Avoid jargon. Use numbered steps for procedures. Length: Comprehensive (5000+ words).

Quick Start Guide

"Create a 2-page Quick Start Guide for [product]. Goal: Get user from zero to first success in under 10 minutes. Structure: 1) What you'll accomplish, 2) Prerequisites (what you need), 3) Step-by-step (numbered, max 7 steps), 4) Verification (how to confirm it worked), 5) Next steps (link to full docs). Format: Simple language, each step is ONE action, include expected outcomes after each step."

Application 4: Process Documentation

Standard Operating Procedure (SOP)

SOP Generation Prompt:

"Create a Standard Operating Procedure document for [process]. Audience: [team members who will execute this].

Format:
- Title and version number
- Purpose (why this process exists)
- Scope (when to use it, when not to)
- Responsibilities (who does what)
- Materials/tools needed
- Procedure (numbered steps with substeps, include decision points)
- Quality checks (how to verify success)
- Troubleshooting (what if something goes wrong)
- Related documents

Tone: Authoritative, precise, leave no room for ambiguity. Use imperatives ('Click X', not 'You should click X')."

Workflow Diagrams (Text-Based)

"Describe the workflow for [process] as a text-based flowchart using ASCII or Mermaid diagram syntax. Include: Start/end points, decision diamonds, process boxes, and arrows showing flow. Also provide a written explanation of each step for someone who can't view the diagram."

Documentation Best Practices

Principle How to Prompt for It
Clarity "Use simple language, avoid jargon, explain acronyms on first use"
Structure "Use clear headings, numbered lists for steps, bullet points for features"
Examples "Include code examples for every major concept, show expected output"
Completeness "Cover all parameters, all error cases, all configuration options"
Accessibility "Describe all visual elements, use semantic markdown, provide text alternatives"

Multi-Audience Documentation

Generate Different Versions for Different Audiences:

"I need to document [system/feature]. Generate 3 versions:

1. Executive Summary: High-level overview for decision-makers (1 page, business value focus)
2. End User Guide: How to use it (step-by-step, non-technical)
3. Technical Implementation: For developers (architecture, APIs, code examples)

Keep consistent information but adjust depth and terminology for each audience."

Keeping Documentation Updated

Diff-Based Documentation Updates:

"Here's the current documentation: [paste]. Here's the code change: [paste git diff or description]. Update the documentation to reflect these changes. Identify which sections need updates and rewrite only those sections. Highlight what changed."

Mini Exercise: Document Your Own Code

Exercise Task:

You have this simple function (Python):

def calculate_total(items, tax_rate=0.08, discount=0):
    subtotal = sum(item['price'] * item['qty'] for item in items)
    discounted = subtotal * (1 - discount)
    total = discounted * (1 + tax_rate)
    return round(total, 2)

Your Tasks:

  1. Write a prompt to generate a Google-style docstring
  2. Write a prompt to create a user guide section explaining when/how to use this function
  3. Write a prompt to generate 3 usage examples (basic, with discount, edge case)
Click to see sample solution

Prompt 1 (Docstring):
"Write a Google-style Python docstring for this function. Include: Brief description, detailed explanation of calculation logic, Args section with types and descriptions, Returns section with type and description, example usage. Here's the function: [paste code]"

Prompt 2 (User Guide):
"Write a user guide section explaining this function. Target audience: developers using our library for the first time. Cover: What it calculates, when to use it, what format 'items' should be (structure), how tax_rate and discount work (percentages as decimals), what gets returned. Use clear examples. Tone: friendly but precise."

Prompt 3 (Examples):
"Generate 3 Python code examples for calculate_total function:
1. Basic: 2 items, default tax, no discount
2. With discount: 3 items, default tax, 15% discount
3. Edge case: Empty items list (should return 0.00)

For each example, show: input data structure, function call, expected output, and brief comment explaining what's being demonstrated."

Key Takeaway:

Great documentation explains not just what something does, but why it exists, when to use it, and how to use it correctly. When prompting for docs, specify your audience (users vs. developers), desired format (markdown, docstrings, manuals), and level of detail. Always include examples—they're often more valuable than explanations.
